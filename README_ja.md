# RigRag

[日本語](README_ja.md) | [English](README.md)

![logo](./puuka.png)

RigRagは、フォルダー内に分散して保存されているオフィス文書（PDF、Word、PowerPoint、Excel など）、Markdown、AsciiDoc、Java ソースコード、EPUB などを一括で読み込み、対話形式でローカル LLM と RAG チャットを行えるローカルツールです。本ツールを利用することで、利用者は資料を個別に開いて確認することなく、質問文を入力するだけで関連情報を横断的に取得できます。議事録、仕様書、設計メモ、管理表、ソースコードなど、形式や保管場所が異なる文書を単一の窓口で参照できるため、情報確認に要する時間を削減し、確認漏れの抑制に寄与します。また、一度読み込んだ内容は プロジェクト内（対象ディレクトリ）に保存されるため、継続利用時も効率的に参照可能です。

ローカルファイル群を読み込んでベクトル化し、Ollama 上の埋め込みモデルと生成モデルを使用して RAG チャットを行う Rust 製 CLI です。対象ディレクトリをインデックス化すると、以下のようなことができるようになります。

- 仕様書・議事録・設計メモを横断して「この機能の制約は何か」を即座に確認できる
- `pdf`、`docx`、`pptx`、`xlsx`、`epub`、`md`、`adoc`、`java` など形式が混在した資料群をまとめて検索できる
- 質問に対して、インデックス済みドキュメントを根拠にした回答を対話形式で得られる
- 単発クエリで「要約」「比較」「抜け漏れ確認」をCLIからすぐ実行できる
- プロジェクトごとに `.rigrag` 配下へインデックスを保存し、継続的に再利用できる
- 他言語で書かれた分量の多いドキュメントでも、rigrag との会話を通じて理解を深められる
- インターネット上のシステムに依存せず、完全にローカルな環境のみで実行できるためセキュアに運用しやすい（初回のモデル取得時を除く）
- LLM が複数の計算ツール（足し算、引き算、掛け算、割り算、合計値計算、平均値計算）を利用して回答を生成できる
- LLM が日付取得ツール（今日の日付取得）を利用して回答を生成できる

## 主な機能

- 対象ディレクトリのファイル走査とチャンク分割
- LanceDB へのベクトル保存とインデックス管理
- RAG 検索結果を使った対話チャット
- LLM エージェント形式での回答生成（ツール呼び出し対応）
- 計算ツール（足し算、引き算、掛け算、割り算、合計、平均）
- 日付取得ツール（今日の日付取得）
- ドキュメント検索ツール
- 単発クエリ実行（`-q/--query`）
- 標準入力からのクエリ入力対応（パイプ経由）
- 必要モデルの存在確認とOllama へのダウンロード案内
- LLM が回答を作成するために参照したドキュメントのパスを表示
- マルチスレッド埋め込み処理（スレッド数の設定可能）
- 画像ファイルの処理と説明生成（Vision対応モデル必要）

## 対応ファイル形式

実装で読み込み対象となるファイル拡張子は以下です。

- `md` - Markdown
- `adoc` - AsciiDoc
- `pdf` - PDF
- `xlsx` - Excel（OOXML形式）
- `xls` - Excel（旧形式）
- `docx` - Microsoft Word
- `pptx` - Microsoft PowerPoint
- `epub` - EPUB 電子書籍
- `java` - Java ソースコード
- `txt` - テキストファイル

## 動作要件

- Rust（推奨: `rustup` 経由）
- Ollama（ローカルで起動していること）
- 初回ビルド時に依存クレートを取得できるネットワーク接続

## セキュリティとローカル運用方針

- rigragは、ローカルのOllamaとローカルファイルを前提に動作し、通常運用では外部のオンラインシステムを利用しません
- ドキュメントのインデックスデータは実行対象ディレクトリ配下の `.rigrag/` に保存されます
- インターネット接続が必要なのは、主に初回セットアップ時（依存クレート取得、Ollamaモデル取得）です
- いったん必要なモデルをローカルに揃えれば、オフライン環境での運用が可能です

## Ollamaのインストール方法

### 1. Windows

方法A（公式インストーラー）:

1. `https://ollama.com/download` にアクセス
2. Windows版インストーラーをダウンロード
3. インストーラーを実行

方法B（winget）:

```powershell
winget install Ollama.Ollama
```

### 2. macOS

方法A（公式インストーラー）:

1. `https://ollama.com/download` からmacOS版を取得
2. インストール後、Ollamaを起動

方法B（Homebrew）:

```bash
brew install --cask ollama
```

### 3. Linux

```bash
curl -fsSL https://ollama.com/install.sh | sh
```

### 4. インストール後の確認

```bash
ollama --version
```

Ollamaを起動し、APIエンドポイント（デフォルト: `http://localhost:11434`）にアクセスできる状態にしてください。

## モデル準備

このプロジェクトのデフォルト設定:

- **生成モデル**：`qwen3.5` - 高精度な日本語対応モデル
- **埋め込みモデル**：`nomic-embed-text-v2-moe` - 高性能な多言語埋め込みモデル

### モデルのダウンロード

アプリ起動時に、指定モデルがローカルにない場合はダウンロードするかを対話的に確認します。

手動で事前取得する場合の例:

```bash
ollama pull qwen3.5
ollama pull nomic-embed-text-v2-moe
```

### 別のモデルを使用する

生成モデルや埋め込みモデルを変更するには、コマンドラインオプションで指定します:

```bash
# 異なる生成モデルを指定
cargo run -- . --model llama3

# 異なる埋め込みモデルを指定
cargo run -- . --embedding-model mxbai-embed-large

# 両方変更
cargo run -- . -m mistral -e mxbai-embed-large
```

### モデル選択の注意点

- **tool calling サポート**：計算ツールを利用するには、モデルが tool calling に対応している必要があります（`qwen*`, `mistral*` など多くが対応）
- **Vision サポート**：`--picture` で画像処理を有効にする場合は、vision 能力を持つモデルが必要（例: `llava`）
- **言語対応**：処理対象のドキュメント言語に対応したモデルを選択（例：日本語なら日本語対応モデル）

## 使い方

### 1. 初回インデックス作成

対象ディレクトリをインデックス化（ベクトル化）します。初回起動時はこれを実行する必要があります。

```bash
cargo run -- <対象ディレクトリ> --init
```

例:

```bash
cargo run -- . --init
```

### 2. 対話モード

チャットウィンドウで対話形式で質問できます。会話履歴が保持され、前の質問の文脈を踏まえた回答が得られます。

```bash
cargo run -- <対象ディレクトリ>
```

起動後はプロンプトに質問を入力します。以下のコマンドが利用できます：
- `exit` - 終了
- `clear` - 会話履歴をクリア

LLM がツール呼び出しをサポートしている場合、必要に応じて計算ツールや検索ツールが自動的に呼び出されます。

### 3. 単発クエリ

`-q/--query` オプションで質問を指定すると、結果を表示して終了します。バッチ処理に向いています。

```bash
cargo run -- <対象ディレクトリ> -q "質問内容"
```

例:

```bash
cargo run -- . -q "このフォルダの設計方針を要約して"
```

### 4. パイプ経由でのクエリ入力

標準入力から質問を渡すこともできます。スクリプト内での利用や他のツールの出力を入力とする場合に有用です。

```bash
echo "ドキュメントの主な内容は？" | cargo run -- .
```

または

```bash
cat questions.txt | cargo run -- .
```

## 主要オプション

- `--init` : インデックスを作り直す
- `-p, --picture` : 画像処理を有効化（利用モデルに vision 能力が必要）
- `-q, --query <TEXT>` : 単発クエリ
- `-u, --url <URL>` : Ollama API URL（デフォルト `http://localhost:11434`）
- `-e, --embedding-model <NAME>` : 埋め込みモデル（デフォルト `nomic-embed-text-v2-moe`）
- `--embedding-thread <N>` : 埋め込み処理のスレッド数（デフォルト `4`）
- `-m, --model <NAME>` : 生成モデル（デフォルト `qwen3.5`）
- `-s, --sample <N>` : RAG検索で取得するコンテキスト数
- `-t, --temperature <VALUE>` : 生成温度（0.0～2.0、デフォルト未指定時はモデルのデフォルト）
- `-x, --exclude <EXT1,EXT2,...>` : 指定した拡張子を処理から除外（カンマ区切り）
- `-v, --verbose` : 詳細表示
- `--single-thread` : 単一スレッドで処理

## データ保存先

実行対象ディレクトリ配下に `.rigrag/` ディレクトリが作成され、以下が保存されます。

- `config.toml` - アプリケーション設定ファイル
- `lancedb-store/` - LanceDB ベクトルデータベース（埋め込み済みドキュメント）

## ツール機能

rigrag は LLM エージェント形式で動作し、以下の組み込みツールを使用できます。

### 計算ツール

LLM が必要に応じて以下のツールを呼び出して計算を実行できます（ツール呼び出し対応モデルが必要）。

- **足し算** - 2つの数値を加算
- **引き算** - 2つの数値を減算
- **掛け算** - 2つの数値を乗算
- **割り算** - 2つの数値を除算（0での除算を防止）
- **合計** - 複数の数値の合計を計算
- **平均** - 複数の数値の平均を計算

### 検索ツール

- **ドキュメント検索** - インデックス済みドキュメント内でベクトル検索を実行し、関連するコンテキストを取得

## トラブルシュート

### Ollama に接続できない場合

- Ollama が起動しているか確認してください
- `ollama serve` で明示的に起動
- `--url` で正しいエンドポイントを指定（デフォルト: `http://localhost:11434`）
- Ollama のログで接続エラーの詳細を確認

### モデル能力エラー

エラーメッセージで以下のような capability 不足が示される場合：

- **embedding/completion 能力なし**: 指定モデルがその能力を保有していない
  - 別のモデルを `--embedding-model` や `--model` で指定
  - 例: `ollama pull nomic-embed-text-v2-moe`
  
- **vision 能力が必要**：`--picture` を使用したが、モデルが vision 对応していない
  - `--picture` メモリを外すか、vision 対応モデルに変更
  - 例: `ollama pull llava` など vision 対応モデルを使用

### ツール呼び出しが機能しない

ツール機能を使用するには、Ollama モデルが tool calling をサポートしている必要があります。

- モデルが tool calling に対応していない場合、計算ツールは呼び出されず、通常の回答のみが生成されます
- Tool calling 対応モデル: `qwen*`, `mistral*` など多くの言語モデルが対応
- モデル情報は起動時に表示される `support_tools: true/false` で確認可能

### インデックス作成が遅い

- CPU 並行数がボトルネックの可能性
  - `--embedding-thread N` で埋め込みスレッド数を調整
  - CPU コア数に応じて最適化（デフォルト: 4）
    - Ollama のモデルサイズやシステムスペックによって最適なスレッド数は異なります  
- 大量のテキスト処理の場合は `--single-thread` でメモリ圧力を軽減

### ドキュメント解析エラー

特定のファイル形式でエラーが出る場合：

- `-x/--exclude` で問題ファイルの拡張子を除外
- 例: `rigrag . -x pdf,xlsx` - PDF と Excel ファイルをスキップ
- 該当ファイルの形式が正しいか確認（破損ファイルなど）

### メモリ不足

大規模なドキュメント群を処理する場合：

- `--single-thread` で並行処理を無効化
- `--embedding-thread 2` など スレッド数を減らす
- インデックス更新時は `--init` で一度リセット後、必要なファイルのみに絞る（`--exclude` で除外）
